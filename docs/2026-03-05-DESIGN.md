# Game design

## Scope

This document sketches the MVP design for the core `game` domain.

It covers:

- the domain model
- the state machine
- the main service methods
- the key invariants

It does not try to solve every edge case. The goal is a small, deterministic model that is easy to build and debug.

## Product assumptions

These assumptions came from the design discussion:

- A user creates an account, then creates a `game`.
- A `game` is a competition with `n + 1` rounds and `n + 1` players.
- The creator drafts the game, defines its rounds, invites players, then manually starts the game.
- Only one round is active at a time.
- Future rounds are editable. The current round and past rounds are not.
- Players may join or leave after the game starts.
- Joining after start is invite-only and controlled by the creator.
- A player who joins during an active submission or voting period may participate immediately if the period is still open.
- A player who leaves during an active period keeps any submission or ballot already made for that period, but cannot affect future rounds.
- Each round has a `theme` and optional `description`.
- During submission, each active player may submit one Spotify track URL and an optional note.
- During voting, each active player must assign exactly 3 stars, one each to 3 distinct submissions.
- Self-voting is not allowed.
- Ties are allowed.
- The round winner is the player or players with the most stars in that round.
- The champion is the player with the most accumulated stars across the full game.
- Round timing is defined by game-level durations, not hand-entered per-round deadlines.

## Design goals

The MVP should:

- keep transitions deterministic
- avoid scattered permission checks
- make one service responsible for advancing state
- store actions as records, not status flags
- stay simple enough to change later

The main design choice is to model the game as an aggregate with an explicit round state machine.

## Core model

### Aggregate

`game` is the top-level aggregate.

Child records:

- `player`
- `round`
- `submission`
- `ballot`
- `star`

### Why this shape

The domain has two kinds of information:

1. **State**: what phase the game is in now
2. **Actions**: what players already did

Those should stay separate.

Examples:

- `round.phase = 'submitting'` says what is allowed now
- a `submission` row says what a player did
- a `ballot` plus 3 `star` rows says how a player voted

This is cleaner than storing flags such as `has_submitted` or `has_voted` on players.

## Table sketch

### `user`

Basic account record.

Suggested fields:

- `id` `varchar(21)` primary key, generated with NanoID
- `email`
- `name`
- `created_at`
- `updated_at`

### `game`

Top-level record and source of timing defaults.

Suggested fields:

- `id`
- `name`
- `description`
- `state` — `draft | active | completed`
- `submission_window_days`
- `voting_window_days`
- `started_at`
- `completed_at`
- `created_at`
- `updated_at`

Notes:

- Use NanoID instead of UUID for cleaner URLs.
- Do not store `current_round_number`; derive it from the round in an actionable phase.
- Do not store `creator_user_id`; the creator is the `player` row with `role = 'creator'`.

### `player`

A player's participation in one game.

Suggested fields:

- `id`
- `game_id`
- `user_id`
- `role` — `creator | player`
- `status` — `invited | active | left | removed`
- `joined_at`
- `ended_at`
- `created_at`
- `updated_at`

Notes:

- One row per `(game_id, user_id)` in MVP.
- Leaving is final in MVP. Rejoin history can be added later if needed.
- `player` is the source of truth for current eligibility.

### `round`

A round definition and its concrete runtime timing.

Suggested fields:

- `id`
- `game_id`
- `number`
- `theme`
- `description`
- `phase` — `pending | submitting | voting | scored`
- `submission_opens_at`
- `submission_closes_at`
- `voting_opens_at`
- `voting_closes_at`
- `spotify_playlist_id`
- `created_at`
- `updated_at`

Notes:

- Game-level durations define the windows.
- Concrete timestamps are still stored on the round when it opens.
- This keeps transitions deterministic and allows extensions without recomputing every future round.

### `submission`

One player's track for one round.

Suggested fields:

- `id`
- `round_id`
- `player_id`
- `spotify_track_url`
- `note` nullable
- `submitted_at`
- `created_at`
- `updated_at`

Notes:

- One row per `(round_id, player_id)`.
- During submission, the same row may be updated until the window closes.

### `ballot`

One player's vote for one round.

Suggested fields:

- `id`
- `round_id`
- `player_id`
- `cast_at`
- `created_at`
- `updated_at`

Notes:

- One row per `(round_id, player_id)`.
- During voting, the ballot may be updated until the window closes.

### `star`

One star allocated to one submission within a ballot.

Suggested fields:

- `ballot_id`
- `submission_id`

Notes:

- A ballot must have exactly 3 `star` rows.
- A ballot may not include the same `submission_id` twice.
- A `star` is implicitly worth 1 point, so no `value` column is needed.

## Naming rules

This design uses:

- singular table names
- short, domain-first names
- `player` instead of `membership`
- `star` instead of `ballot_star`

## State machine

### Game states

`game.state`:

- `draft`
- `active`
- `completed`

Transitions:

- `draft --start_game--> active`
- `active --complete_game--> completed`

### Round phases

`round.phase`:

- `pending`
- `submitting`
- `voting`
- `scored`

Transitions:

- `pending --open_submission--> submitting`
- `submitting --close_submission--> voting`
- `submitting --close_submission_with_too_few_submissions--> scored`
- `voting --close_voting--> scored`

### Sequencing rule

Only one round may be actionable at a time.

In practice, that means only one round may be in either:

- `submitting`, or
- `voting`

All other rounds are either:

- `pending`, or
- `scored`

### Timing rule

The game stores:

- `submission_window_days`
- `voting_window_days`

When a round opens for submission:

- `submission_opens_at = now()`
- `submission_closes_at = now() + submission_window_days`

When a round opens for voting:

- `voting_opens_at = now()`
- `voting_closes_at = now() + voting_window_days`

This is the recommended MVP compromise:

- creators configure durations once at the game level
- the system stores concrete timestamps on the active round

## Capability model

The app should not scatter permission checks across handlers and UI code.

Instead, the domain should compute a set of allowed actions.

Suggested API:

```ts
type Action =
  | "start_game"
  | "edit_game"
  | "edit_future_round"
  | "invite_player"
  | "accept_invite"
  | "leave_game"
  | "remove_player"
  | "submit_song"
  | "delete_submission"
  | "cast_ballot"
  | "transition_round"
  | "extend_active_window";

type CapabilityContext = {
  game: Game;
  activeRound: Round | null;
  actorPlayer: Player | null;
  actorUserId: string;
  now: Date;
};

function getAllowedActions(ctx: CapabilityContext): ReadonlySet<Action>;
```

MVP decision:

- return only allowed actions
- do not store or return deny reasons

### Why a set of actions

This is cleaner than a large object of booleans.

Benefits:

- one domain vocabulary for permissions
- easier to test
- easier to extend
- avoids a long list of irrelevant `false` values

A UI can always wrap it with helper functions like `can('submit_song')`.

## Main service methods

The public service API should stay small and command-oriented.

```ts
createGame(actorId, input);
updateGame(actorId, gameId, patch);
addRound(actorId, gameId, input);
updateFutureRound(actorId, gameId, roundId, patch);
invitePlayer(actorId, gameId, userId);
acceptInvite(actorId, gameId);
leaveGame(actorId, gameId);
removePlayer(actorId, gameId, playerId);

startGame(actorId, gameId);

saveSubmission(actorId, gameId, roundId, {
  spotifyTrackUrl,
  note,
});

deleteSubmission(actorId, gameId, roundId);

upsertBallot(actorId, gameId, roundId, {
  submissionIds: [a, b, c],
});

transitionGame(gameId, now);

extendActiveWindow(actorId, gameId, roundId, {
  phase: "submitting" | "voting",
  deltaDays,
});
```

Notes:

- `saveSubmission` is better than separate create/update methods.
- `upsertBallot` is better than a one-shot cast API, because players may revise votes until the window closes.
- `extendActiveWindow` is optional for MVP, but the schema supports it cleanly.
- `startGame` must set `game.state = 'active'` and open round 1 in the same transaction. An active game with no actionable round is invalid.

## Command flow

Every write should follow the same structure inside one transaction:

1. begin a transaction
2. lock the game and any relevant round or player rows
3. call `transitionGame(tx, gameId, now)`
4. load the capability context from the same transaction
5. compute allowed actions
6. assert the needed action is present
7. assert command-specific invariants
8. write rows

This avoids a TOCTOU race where permissions are computed against stale state and the round or player changes before the write commits.

Pseudo shape:

```ts
async function command(...) {
  return db.transaction(async (tx) => {
    await lockGameAggregate(tx, { gameId, actorId, roundId });
    await transitionGame(tx, gameId, now);

    const ctx = await loadCapabilityContext(tx, gameId, actorId, now);
    const actions = getAllowedActions(ctx);

    assert(actions.has(requiredAction));
    assertDomainInvariants(ctx, ...);

    return writeCommand(tx, ...);
  });
}
```

This keeps transition, authorization, invariant checks, and writes on the same locked snapshot.

## Transition service

`transitionGame(tx, gameId, now)` should be the only method that advances state.

Suggested internal helpers:

```ts
openFirstRound(tx, gameId, now);
closeSubmissionAndOpenVoting(tx, gameId, roundId, now);
closeSubmissionAsVoid(tx, gameId, roundId, now);
scoreRound(tx, gameId, roundId, now);
openNextRoundOrCompleteGame(tx, gameId, now);
completeGame(tx, gameId, now);
```

### `startGame` semantics

`startGame` should not only flip `game.state`.

It must, in one transaction:

- lock the game and round rows
- verify the start invariants
- set `game.state = 'active'`
- set `game.started_at = now()`
- move round 1 from `pending` to `submitting`
- set round 1 submission timestamps

That keeps the invariant simple: a healthy active game always has exactly one actionable round.

### Transition algorithm

```ts
transitionGame(tx, gameId, now) {
  lock game + actionable round rows

  const game = getGameForUpdate(tx, gameId)

  if (game.state === 'draft' || game.state === 'completed') return noOp

  const round = getActiveRound(tx, gameId)

  if (!round) {
    const nextRound = getFirstPendingRound(tx, gameId)

    if (!nextRound) {
      return completeGame(tx, gameId, now)
    }

    return openFirstRound(tx, gameId, now)
  }

  if (round.phase === 'submitting' && now >= round.submissionClosesAt) {
    if (submissionCount(tx, round.id) < 4) {
      return closeSubmissionAsVoid(tx, gameId, round.id, now)
    }

    return closeSubmissionAndOpenVoting(tx, gameId, round.id, now)
  }

  if (round.phase === 'voting' && now >= round.votingClosesAt) {
    scoreRound(tx, gameId, round.id, now)
    return openNextRoundOrCompleteGame(tx, gameId, now)
  }

  return noOp
}
```

In healthy state, `!round` should be rare because `startGame` opens round 1 atomically and later transitions always open the next round or complete the game. The defensive branch exists so the system can recover from partial writes or manual data repair.

### Why the `4 submissions` rule exists

Given the chosen voting rules:

- exactly 3 stars are required
- each star must target a different submission
- self-voting is forbidden

A player needs at least 3 other submissions to vote for.

That means voting only works if a round has at least 4 submissions.

MVP recommendation:

- require at least 4 active players to start a game
- if a round closes with fewer than 4 submissions, mark it `scored` with no winners and move on

This is a product choice, not a database constraint. It can be revisited later.

## Spotify integration

For MVP, Spotify playlist creation should happen in the submission-to-voting transition path.

Suggested flow:

1. collect round submissions
2. create the Spotify playlist
3. store `spotify_playlist_id`
4. switch the round to `voting`
5. write voting timestamps

MVP shortcut:

- perform Spotify work inline in the transition path
- commit the phase change only after Spotify succeeds

Cost of this shortcut:

- if Spotify succeeds but the database transaction fails, an orphan playlist may be left behind

Replacement condition:

- if retries, observability, or failure recovery become important, move this to an outbox or background job and add an intermediate phase such as `preparing_playlist`

## Key invariants

### Database-enforced invariants

These should be enforced with unique indexes and check constraints.

- one `player` per `(game_id, user_id)`
- one creator per game
- one `round` per `(game_id, number)`
- one `submission` per `(round_id, player_id)`
- one `ballot` per `(round_id, player_id)`
- one `star` per `(ballot_id, submission_id)`
- positive `submission_window_days`
- positive `voting_window_days`
- positive `round.number`

### Service-enforced invariants

These belong in domain code for MVP.

- only active players may submit or vote
- only the active round may accept writes
- players may only edit their own submission or ballot
- self-voting is forbidden
- a ballot must target exactly 3 distinct submissions
- all 3 starred submissions must belong to the same round as the ballot
- the current and past rounds are immutable
- only future rounds are editable once the game is active

## Suggested Drizzle posture

Use Postgres with Drizzle as a thin schema and query layer.

Recommended posture:

- let the database guard record shape and uniqueness
- let the service layer guard gameplay rules
- keep transitions in explicit application code
- prefer transactions over clever triggers in MVP

This is the simplest model that still keeps the hard parts centralized.

## Open questions for later

These are intentionally out of scope for MVP, but the design leaves room for them.

- Should a player be able to rejoin after leaving?
- Should a creator be able to reopen a closed submission or voting period?
- Should games support per-round timing overrides?
- Should there be explicit invite tokens or join codes?
- Should the app snapshot display names at submission time for audit/history?
- Should a round with too few submissions be skipped, cancelled, or extended instead of voided?

## Summary

The recommended MVP design is:

- `game` as the aggregate
- singular, simple table names
- `player` as the participation record
- one explicit round state machine
- game-level duration settings with per-round derived timestamps
- actions stored as rows
- permissions returned as a `Set<Action>`
- one transition service responsible for advancing phases

This keeps the domain deterministic, keeps permission logic centralized, and avoids brittle ad hoc checks throughout the app.
